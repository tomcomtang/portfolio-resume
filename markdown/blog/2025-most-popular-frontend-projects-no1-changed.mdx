在 2025 年的 JavaScript Rising Stars 榜单里，“热度”不只是体现在抽象的趋势词上，更体现在具体产品的迭代速度与工程化落地能力。下面这篇文章，我会把榜单中最值得关注的变化做一次结构化复盘：从最受欢迎的项目切入，再扩展到框架、生态、全栈、工具、AI 和移动端。

## 前言：这份榜单到底在看什么

JavaScript Rising Stars 会根据 2025 年“新增星标数”来建立多个分榜单。也就是说，你看到的不只是“过去很强”，而是“最近更容易被开发者持续选择与关注”。榜单覆盖的维度包括：最受欢迎项目、前端框架、React 生态、后端/全栈、构建工具、人工智能、移动端、Vue 生态、状态管理、CSS-in-JS、组件库、测试、桌面端、静态站点以及 GraphQL。

## 最受欢迎的前端项目

### 2.1 总冠军：[`n8n`](https://github.com/n8n-io/n8n)

今年的总冠军是 [`n8n`](https://github.com/n8n-io/n8n)。它在一年内收获了极其可观的星标增量（文中提到超过 112000）。和很多“能用就行”的自动化工具不同，[`n8n`](https://github.com/n8n-io/n8n) 的优势在于：它把“工作流自动化”与“原生 AI 能力”结合得比较紧密，并且提供了更适合工程团队协作的可视化串联方式。

同样值得注意的是，[`n8n`](https://github.com/n8n-io/n8n) 之外的前十项目里，还出现了多项与 AI 直接相关的内容，例如：

- `Onlook`：面向 React 应用的 AI 优先视觉编辑能力
- `Dyad`：免费、本地化、开源的 AI 应用构建器（被视为 v0/lovable/Bolt 的替代）
- `Stagehand`：面向自动化的 AI 驱动浏览器任务

### 2.2 亚军：[`React Bits`](https://github.com/DavidHDev/react-bits)

React Bits 更像是“面向生产的灵感库”。它提供了一套风格统一、适合直接嵌入页面的 React 动画组件（包括背景效果、文本动画和卡片等），目标很明确：让你在不牺牲可维护性的前提下，快速做出更有质感的前端展示。

有意思的是它与 [`shadcn/ui`](https://github.com/shadcn-ui/ui) 的生态联动较深：你可以通过相应方式把组件注册到自己的项目里，甚至还带了一个后台工作室，用来调参并导出可复用的代码片段。

### 2.3 季军：[`shadcn-ui`](https://github.com/shadcn-ui/ui)

季军落在 [`shadcn-ui`](https://github.com/shadcn-ui/ui)。它的核心价值不只是“组件好看”，而是更偏工程细节：一致的交互体验、较好的可访问性，以及对键盘操作与语义结构的重视。

同时，它还融合了诸多底层优秀实践（例如 Radix UI、TanStack Table 等无头能力）。从工程角度看，它很好地在“开箱即用”和“可深度定制”之间找到了平衡点。

## 前端框架：React 再次回到核心位置

原文的观点很直接：今年 React 重新拿回了“前端框架榜首”。原因除了社区热度外，更重要的是它正在向“服务器优先”的开发范式迁移。

例如 React 19 提供了与 Activity API 相关的重要改进，以及用于管理用户事件的增强型 hooks。与此同时，React 的 effects 生态也再次成为讨论焦点：文中提到，`Cloudflare` 因为错误使用 `useEffect` 导致无限调用 API，进而引发严重的服务中断与 DDoS 风险。

更大的结构性变化来自 React Server Components：React 以 RSC、server actions / server functions、流式传输等方式，把功能与边界重新组织。但这类变化也带来新的风险面。文中提到一次围绕 RSC 的远程代码执行（RCE）漏洞（需要紧急发布补丁）。

除了 React，`Ripple` 也被重点提及：它被描述为融合 React、Solid 和 Svelte 思路的新 UI 框架，强调响应式原语、组件化架构以及模板语法的清晰可控。

在“状态表达”方面，`Svelte 5` 的运行机制（例如更标准化的响应式方式）也让它在榜单中延续了强势表现：用新的响应式模型来降低心智成本，让复杂 UI 更容易写对。

## React 生态：强服务器能力与开发简洁性的矛盾

原文把 2025 React 生态的转折归结为一个长期矛盾：更强的服务器能力与保持客户端开发简单可预测之间存在冲突。

在 `Next.js` 的推动下，React 体系向 RSC、服务器函数和流式传输发展，让性能和架构出现新可能。但也要求开发者必须更清楚地理解：客户端与服务器的边界在哪里、数据生命周期如何流转、渲染阶段各自承担什么责任。

对生态的评价出现分歧。一部分人认为这是真正的自然演进，另一部分人则担心复杂度增加、抽象层过厚会带来更多“难以预期”的工程风险。文中还提到围绕安全事件的讨论进一步加剧了这种分歧。

在这种背景下，`TanStack Start` 受到关注：它更偏向“以客户端为中心的同构视角”，强调清晰性与类型安全，并希望把讨论从框架争论拉回到“工程师应该接受多少抽象”这一更实际的议题上。

## 后端 / 全栈：从“拼装能力”到“统一抽象”

全栈榜单里，`Motia` 被描述为一种范式转变：它试图把过去需要多个独立框架组合才能完成的能力，整合进同一个系统里。你不再需要分别处理 API、后台任务、队列、工作流、数据流以及 AI 代理等拼装问题，而是以统一的方式组织它们。

它的关键抽象是“步骤（Steps）”：每一步既包含触发与运行配置，也包含具体处理逻辑；当改变步骤类型时，同一套模式可以复用到 API 端点、事件处理程序或定时任务等场景。

在工程实现上，它支持使用 TypeScript 或 Python 编写，并配套了可观测性与可视化工作台（Workbench），使调试与运行状态的跟踪更直观，同时内置状态管理与流式能力。

原文还提到一些延续与互换的趋势：例如 `Payload` 在 `Next.js` 混合型无头 CMS 的基础上持续演进，并且被 Figma 收购；`Next.js 16` 引入缓存相关能力，让缓存机制更“显式”；`Astro` 仍保持高位；`Hono` 以轻量核心与中间件生态成为现代 Web 服务的重要选择；以及 `TanStack Start` 在全栈方向继续崛起。

## 构建工具：Bun、Oxc/Rolldown 与测试生态的变化

工具榜单的亮点主要集中在两个点：更“全栈式”的工具链整合，以及与工程质量相关的测试/运行模式更新。

文中把 `Bun` 放在第一位：它被认为是一体化的 JavaScript 工具包，持续增强对 Node.js 的兼容性并推进性能改进；同时它还在年底被 Anthropic 收购，让后续发展更值得期待。

在“新一代打包与编译”方向，`void(0)` 的 `Oxc` 与 `Rolldown` 被反复提到：它们让 Vite 生态获得新一轮的工具扩展，同时 `Vitest` 也推出了更受期待的功能之一——浏览器模式（browser mode）。

`Rspack` 也出现在榜单讨论中：作为 Rust 生态驱动的打包工具，它被认为速度更快、与现有生产工作流适配度高，因此被更广泛采用。

此外，`Next.js` 生态切换到 Turbopack 也被点名：打包工具的变化直接影响开发与反馈速度；而 TypeScript 相关的“Go 重写版”预告则被视为未来基础设施的重要信号（文中指出 TS6 作为过渡版）。

## 人工智能：从“聊天”转向“工作流引擎”

原文认为，聊天机器人时代正在结束。更重要的是：开发者开始把注意力从“提示词能不能更好”转向“工作流能不能直接自动化”。

以 [`n8n`](https://github.com/n8n-io/n8n) 为例，文中提到它预计新增用户规模，并且在连接模型与构建代理工具方面正在成为更常见的选择。与此同时，还有 Flowise、Mastra、Stagehand 等公司在自动化与代理工作流上持续进展。

文中也强调了思路：与其不断追问“如何让 LLM 回答得更好”，不如优先考虑“哪些工作流可以由 AI 全流程接管”。例如：

- 用 [`n8n`](https://github.com/n8n-io/n8n) 或 `Flowise` 构建事件触发、推理选项并自动执行的程序
- 用 `Mastra` 做跨工具的平台协调代理
- 用 `Stagehand` 自动化浏览器任务

最后原文还特别提到一个被认为可能更关键的方向：`code mode`，并将其视为自 MCP 以来该领域更重大的突破之一。

## 移动端：React Native 失去榜首，被新框架改写

移动端榜单里，React Native 以及 Expo 今年未能继续拿到第一。取而代之的是 `Valdi` 与 `Lynx` 两个新晋框架。

`Valdi` 被描述为更偏“轻量、延迟加载与可扩展”的路线：它希望让用户可以更有选择性地启用新功能，而不会造成明显性能代价；同时它借鉴了 Web 技术表达方式，支持 TypeScript、JSX、Flexbox 等开发体验。

`Lynx` 则更偏向“交互体验”：文中强调它采用双线程架构以避免单线程瓶颈，并提供类似 Web 抽象层的能力，默认推荐使用更自然的 React 抽象。

文中还提到移动端并不只有这一条变化线：例如排名第四的 `Dioxus` 被视为面向更优秀的 Flutter 的挑战者；尽管当前默认使用系统 WebView，但其长期目标是稳定 Blitz，并通过 wgpu 使用原生图形 API（例如 Vulkan 和 Metal）进行渲染。

## 结语：把榜单信息转成你的工程决策

如果把这些榜单当作“趋势雷达”，真正有效的做法不是跟风，而是把它们落到可执行的工程选择上：看清楚每个项目在解决什么问题、工程边界是什么、风险点在哪里，然后用小成本实验验证它是否适合你的生产场景。

这就是这份榜单复盘我想强调的重点：变化很多，但方向往往很明确——工具链更快、更一致，框架更向服务器与边界清晰化发展，AI 更像“能力组件”，最终回到工程落地与可维护性。
